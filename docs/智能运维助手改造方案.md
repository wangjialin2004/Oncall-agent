# 智能运维助手改造方案

## 目标

将当前项目从“可演示的 Agent Demo”改造成一个结构清晰、数据真实、检索可控、后续可扩展的智能运维助手。

改造重点不是单纯让大模型多调用几个工具，而是补齐以下能力：

- 明确被监控对象
- 统一 RAG 检索入口
- 支持 BM25 + 语义向量混合检索
- 让 MCP Server 从本地假数据过渡到真实数据适配
- 让 RAG Agent 和 AIOps Agent 有清晰调度关系
- 增加业务记忆层和诊断证据链
- 补齐测试、健康检查和工程化边界

## 当前主要问题

### 1. 缺少真实监控对象

当前 `mcp_servers/monitor_server.py` 中的 CPU、内存数据是代码动态生成的，并不来自真实服务、真实机器或真实监控平台。

这导致 AIOps Agent 虽然能执行“查指标 -> 查日志 -> 生成报告”的流程，但诊断对象并不真实。

建议先把监控对象定义为：

> 智能运维助手自身运行环境，包括 FastAPI 服务、MCP 服务、Milvus 向量库、LLM 调用链路、RAG 索引检索流程和本机基础资源。

这样即使没有外部业务系统，也能让 Agent 诊断真实对象。

### 2. MCP Server 仍是本地数据适配层

当前两个 MCP Server 的定位：

- `cls_server.py`：提供日志查询工具
- `monitor_server.py`：提供监控指标工具

问题：

- `cls_server.py` 目前只返回本地 topic 和动态日志
- `monitor_server.py` 目前只生成 CPU/内存曲线
- MCP 工具没有统一 Provider 抽象
- 工具返回结果没有证据 ID、耗时、数据来源、错误类型等审计字段

### 3. RAG 检索入口重复

当前项目里存在两套检索方式：

- `knowledge_tool.py` 直接调用 `vector_store_manager.get_vector_store().as_retriever()`
- `vector_search_service.py` 自己调用 Milvus `collection.search()`

实际被 Agent 使用的是 `knowledge_tool.py` 里的 LangChain retriever。

`vector_search_service.py` 当前没有被业务调用，属于未完成的检索抽象。

### 4. Milvus 只做语义向量检索

当前 Milvus collection schema 只有：

- `id`
- `vector`
- `content`
- `metadata`

因此当前检索是纯 dense vector semantic search。

运维知识库中经常出现错误码、异常类名、服务名、配置项、命令、日志关键字，这些内容只靠语义向量并不稳定。

Milvus 2.5 支持 BM25 全文检索和 dense + sparse hybrid search，当前项目没有使用。

### 5. RAG Agent 和 AIOps Agent 没有统一调度

当前链路是用户选择入口：

- 普通聊天 -> `/api/chat` 或 `/api/chat_stream` -> `RagAgentService`
- AI Ops 按钮 -> `/api/aiops` -> `AIOpsService`

两者并不是由一个 Router Agent 自动调度。

AIOps Planner 会调用 `retrieve_knowledge` 查知识库，但它不是调用 `RagAgentService`，只是复用知识检索工具。

### 6. 记忆层还停留在进程内状态

当前使用 `MemorySaver` 保存会话状态。

问题：

- 进程重启后丢失
- 只能保存 LangGraph 运行状态
- 没有诊断历史、工具调用证据、用户反馈、历史根因等业务记忆

### 7. 文档清洗和切分较基础

当前 `document_splitter_service.py` 已完成 Markdown/text 切分，但更像基础切分：

- 没有文档清洗 pipeline
- 没有标题层级强化
- 没有 chunk 质量检查
- 没有去重
- 没有文档版本
- 没有索引任务状态

## 改造原则

### 1. 保持 Agent 层稳定，优先改数据和检索层

先不要大改 Planner/Executor/Replanner。

优先让工具返回真实可信的数据，让 RAG 检索更可靠。

### 2. 统一入口，减少重复实现

RAG 检索应该只有一个核心入口：

```text
knowledge_tool
  -> vector_search_service
  -> vector_store_manager / Milvus
```

`knowledge_tool.py` 负责暴露给 Agent。

`vector_search_service.py` 负责检索策略。

`vector_store_manager.py` 负责 Milvus 连接、collection、schema 和底层存储。

### 3. 将本地数据变成可替换 Provider

MCP Server 不应该直接写死数据生成逻辑。

建议改成：

```text
MCP Tool
  -> Provider Interface
  -> LocalProvider / SelfMonitorProvider / PrometheusProvider / CLSProvider
```

先实现本机/自身监控 Provider，后续再接真实平台。

### 4. 所有诊断结论必须有证据

Agent 最终报告里应区分：

- 已确认事实
- 推测原因
- 缺失证据
- 建议动作

每条关键结论都应能追溯到工具调用结果或知识库引用。

## 目标架构

```text
Frontend
  -> Chat API
      -> Router Service
          -> RAG Agent
          -> AIOps Agent

RAG Agent
  -> knowledge_tool
      -> vector_search_service
          -> dense search
          -> BM25 sparse search
          -> merge / rerank
          -> formatted references

AIOps Agent
  -> planner
  -> executor
      -> MCP tools
      -> knowledge_tool
  -> replanner
  -> diagnosis memory

MCP Servers
  -> cls_server
      -> LogProvider
  -> monitor_server
      -> MonitorProvider

Memory Layer
  -> session memory
  -> diagnosis case memory
  -> tool evidence memory
  -> user feedback memory
```

## 分阶段改造计划

## 阶段一：明确监控对象，改造 MCP Server

### 目标

让项目不再依赖虚构服务，而是能监控“智能运维助手自身运行环境”。

### 修改内容

#### 1. 新增监控对象定义

新增服务对象：

```text
aiops-assistant-api
cls-mcp-server
monitor-mcp-server
milvus
local-machine
rag-indexer
llm-provider
```

#### 2. 改造 `monitor_server.py`

新增工具：

- `check_api_health()`
- `check_mcp_health()`
- `check_milvus_health()`
- `get_local_resource_usage()`
- `get_service_ports_status()`
- `get_python_processes()`
- `get_disk_usage()`

保留原有 `query_cpu_metrics` 和 `query_memory_metrics`，但内部应改为调用真实本机数据或 Provider。

#### 3. 改造 `cls_server.py`

新增工具：

- `get_recent_app_logs()`
- `search_app_logs(keyword, level, limit)`
- `get_log_files()`

先读取本项目 `logs/` 目录下的应用日志。

### 产出

Agent 可以真实回答：

- FastAPI 是否存活
- MCP 服务是否可访问
- Milvus 是否正常
- 本机 CPU/内存/磁盘是否异常
- 最近应用日志是否有错误

## 阶段二：统一 RAG 检索入口

### 目标

让所有 RAG 查询都经过 `vector_search_service.py`，避免 LangChain retriever 和手写 Milvus search 并存。

### 修改内容

#### 1. 定义统一搜索结果

`SearchResult` 应包含：

```python
id: str
content: str
score: float
source: str
metadata: dict
retrieval_type: str
rank: int
```

#### 2. 改造 `knowledge_tool.py`

从：

```python
vector_store = vector_store_manager.get_vector_store()
retriever = vector_store.as_retriever(...)
docs = retriever.invoke(query)
```

改为：

```python
results = vector_search_service.search(query, top_k=config.rag_top_k)
```

#### 3. 明确职责

```text
vector_embedding_service.py
  只负责 embedding

document_splitter_service.py
  只负责清洗和切分

vector_index_service.py
  只负责索引任务

vector_store_manager.py
  只负责 Milvus 连接和 collection 管理

vector_search_service.py
  只负责检索策略

knowledge_tool.py
  只负责暴露给 Agent
```

### 产出

所有 Agent 检索都走同一个检索服务。

## 阶段三：升级 Milvus 为 BM25 + 语义混合检索

### 目标

让知识库同时支持：

- 语义召回
- 关键词精确召回
- 混合排序

### 当前问题

当前 collection 只有一个 `vector` 字段，无法支持 BM25 sparse vector。

### 目标 schema

建议改为：

```text
id              VARCHAR primary key
content         VARCHAR analyzer enabled
dense_vector    FLOAT_VECTOR(1024)
sparse_vector   SPARSE_FLOAT_VECTOR
metadata        JSON
```

### 检索流程

```text
query
  -> dense embedding
  -> dense search

query
  -> BM25 function
  -> sparse search

dense results + sparse results
  -> score normalize
  -> weighted merge
  -> optional rerank
  -> top_k
```

### 建议权重

初始可以使用：

```text
dense_weight = 0.7
bm25_weight = 0.3
```

对于运维类查询，如果 query 中包含错误码、异常类名、服务名、配置项，可以提高 BM25 权重。

### 注意

这一步通常需要重建 collection 并重新索引文档。

## 阶段四：增强文档处理 pipeline

### 目标

让索引质量可控，而不是简单读取文件后直接切分入库。

### 修改内容

#### 1. 增加文档清洗

处理：

- 多余空行
- 不可见字符
- Markdown 代码块保留
- 标题层级
- 表格内容
- 重复内容

#### 2. 增加 chunk 元数据

每个 chunk 应包含：

```text
source
file_name
file_hash
chunk_id
chunk_index
heading_path
content_type
created_at
```

#### 3. 增加索引任务状态

上传文件后不应只同步索引。

建议增加：

```text
pending
indexing
completed
failed
```

## 阶段五：增加 Router Agent

### 目标

让系统从“用户选择按钮”升级为“系统自动判断意图”。

### 当前情况

```text
用户普通聊天 -> RAG Agent
用户点击 AI Ops -> AIOps Agent
```

### 目标情况

```text
用户输入
  -> Router Agent
      -> RAG 问答
      -> AIOps 诊断
      -> 文件/知识库操作
      -> 追问澄清
```

### 路由规则

先用规则 + LLM 轻量分类即可。

示例：

```text
包含“告警、故障、异常、CPU、内存、日志、诊断、服务不可用”
  -> AIOps Agent

包含“文档、知识库、怎么处理、排查步骤、说明”
  -> RAG Agent

不确定
  -> 追问用户
```

## 阶段六：建设业务记忆层

### 目标

把记忆从“聊天历史”升级为“诊断资产”。

### 需要保存的内容

#### 1. Session Memory

- 用户消息
- AI 回复
- thread_id

#### 2. Diagnosis Case Memory

- case_id
- session_id
- alert/service/env
- plan
- executed_steps
- final_report
- status

#### 3. Tool Evidence Memory

- tool_name
- arguments
- raw_result
- summary
- success
- error
- duration_ms

#### 4. Feedback Memory

- 用户是否认可结论
- 实际根因
- 最终处理方式

### 存储建议

早期可用 SQLite。

后续可切到 PostgreSQL。

长期经验可继续进入 Milvus。

## 阶段七：补齐工程质量

### 测试

至少补充：

- 文档切分测试
- 文件上传索引测试
- 向量检索测试
- RAG tool 格式化测试
- MCP health 工具测试
- AIOps 诊断流程最小测试

### 健康检查

`/health` 应检查：

- FastAPI 自身
- Milvus 连接
- MCP Server 可达性
- LLM 配置是否存在
- RAG collection 是否存在

### 配置

新增配置项：

```text
RAG_RETRIEVAL_MODE=dense|bm25|hybrid
RAG_DENSE_WEIGHT=0.7
RAG_BM25_WEIGHT=0.3
RAG_RERANK_ENABLED=false
MONITOR_TARGET_MODE=self|local|prometheus
LOG_PROVIDER=local|cls
```

## 推荐实施顺序

### 第一步：改 MCP 监控对象

先让 Agent 有真实对象可查。

优先级最高。

### 第二步：统一 RAG 检索入口

把 `knowledge_tool.py` 改为只调用 `vector_search_service.py`。

### 第三步：增强 RAG 检索

先做 dense search 的统一封装，再升级 hybrid search。

### 第四步：增加诊断证据结构

让每个工具调用和最终报告可追溯。

### 第五步：增加 Router Agent

把两个入口统一为一个智能助手入口。

### 第六步：做记忆层和历史诊断

把诊断结果沉淀下来，成为长期能力。

## 不建议立刻做的事情

- 不建议一开始就重写 Planner/Executor/Replanner
- 不建议马上接复杂监控平台
- 不建议直接把所有数据都塞进向量库
- 不建议只靠 Prompt 解决证据链问题
- 不建议继续维护两套 RAG 检索入口

## 最小可落地版本

如果只做一个最小但可信的版本，建议范围是：

1. `monitor_server.py` 改为监控本项目自身
2. `cls_server.py` 改为读取本项目日志
3. `knowledge_tool.py` 统一走 `vector_search_service.py`
4. `vector_search_service.py` 返回标准 `SearchResult`
5. AIOps 报告里展示工具证据摘要
6. `/health` 检查 Milvus、MCP、LLM 配置

完成后，这个项目就不再只是流程演示，而是具备真实诊断对象和可验证检索链路的智能运维助手。
