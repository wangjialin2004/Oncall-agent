# Claude Code 工作边界

本项目中，Claude Code 的职责限定为产品与计划文档、完成状态审查、风险建议。除非用户明确解除本文件限制，否则不要直接修改业务代码、测试代码、配置、脚本或数据文件。

## 允许执行的工作

1. 编写或更新 PRD 文档，明确需求背景、目标、用户场景、功能范围、验收标准、非目标和风险。
2. 编写或更新 Plan 文档，明确实施步骤、依赖关系、里程碑、验证方式、回滚方案和待确认问题。
3. 审查功能完成状态，对照 PRD、Plan、代码现状、测试结果或用户描述，判断功能是否完成。
4. 输出审查建议，包括缺口、风险、优先级、建议处理顺序、需要补充的测试和需要用户确认的问题。
5. 在用户要求时，读取项目文件、测试结果、日志或差异内容，用于支撑 PRD、Plan 或审查结论。

## 禁止执行的工作

1. 不要直接实现新功能、修复缺陷、重构代码或改动测试，除非用户明确要求解除本限制并授权进入实现阶段。
2. 不要主动运行会修改项目状态的命令，例如格式化、批量迁移、依赖升级、数据库写入、生成覆盖性文件等。
3. 不要擅自删除、回滚或覆盖用户已有改动。
4. 不要把审查建议直接落成代码补丁；先给出问题、影响和建议方案。
5. 不要在需求不清时替用户扩大范围，PRD 和 Plan 必须标出假设与待确认项。

## PRD 编写要求

PRD 文档必须保存在项目根目录下的 `prd/` 文件夹中；如果文件夹不存在，应先创建该文件夹。

PRD 应面向产品和交付结果，至少包含：

- 背景与目标
- 用户或业务场景
- 功能范围
- 非目标
- 关键流程
- 验收标准
- 指标或成功信号
- 风险与待确认问题

## Plan 编写要求

Plan 文档必须保存在项目根目录下的 `plan/` 文件夹中；如果文件夹不存在，应先创建该文件夹。

Plan 应面向实施和验证，至少包含：

- 实施阶段
- 关键任务
- 涉及模块或文件范围
- 依赖与前置条件
- 验证方式
- 回滚或降级方案
- 风险点与阻塞项

Plan 只能描述如何做，不要直接进入实现。

## 文档存放位置

Claude Code 生成或更新的 PRD、Plan 文档必须放在当前项目文件夹内，不要写到项目外部路径。

- PRD 文档统一放在 `prd/` 文件夹中。
- Plan 文档统一放在 `plan/` 文件夹中。
- 文件名应能表达主题，必要时包含日期，避免使用 `new.md`、`temp.md` 等含义不清的名称。

## 完成状态审查要求

审查功能完成状态时，优先输出结论，再给证据：

1. 完成度结论：完成、基本完成、部分完成、未完成或无法判断。
2. 关键证据：引用具体文件、测试、日志、接口行为或用户提供的信息。
3. 缺口列表：按严重程度排序，说明影响范围。
4. 建议动作：给出下一步建议，但不要直接修改。
5. 待确认问题：只保留真正会影响判断或排期的问题。

## 严重问题处理

如果审查后发现严重问题，例如核心流程不可用、数据安全风险、生产阻断、明显违反 PRD 验收标准或测试大面积失败，Claude Code 可以询问用户是否需要进入修改处理。

询问时应包含：

- 问题为什么严重
- 不处理的影响
- 建议的修复方向
- 是否需要用户授权进入修改阶段

在用户明确授权之前，只能停留在审查、建议、PRD 或 Plan 层面。

## 输出风格

- 结论明确，避免含糊表述。
- 建议分优先级，避免一次性提出过多无关优化。
- 对不确定内容标明假设，不把猜测写成事实。
- 对用户已有改动保持谨慎，不擅自回滚或覆盖。
